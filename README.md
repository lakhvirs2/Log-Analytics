# HDFS Anomaly Detection – Big Data ML Pipeline

End-to-end anomaly detection system for HDFS (Hadoop Distributed File System) log data,
implemented with **Apache Spark / PySpark MLlib** on a 1.7 GB, ~3.45 million row dataset.

## 1. Project Overview

| Property | Detail |
|---|---|

## 7. ML Models

### Spark MLlib Models

| Model | Features used | Key hyperparameters | Tuning |
|---|---|---|---|
| Logistic Regression | `std_features` | maxIter=100, regParam=0.01 | None |
| Random Forest | `raw_features` | numTrees=100, maxDepth=10 | None |
| GBT | `raw_features` | maxDepth∈{4,6}, stepSize∈{0.05,0.1,0.2} | 3-fold CV |
| Linear SVC | `std_features` | maxIter=100, regParam=0.01 | None |

All models use `class_weight` to handle label imbalance.

### Scikit-learn Baseline

- Logistic Regression on a 2 % stratified sample
- Serves as a quick sanity check for the Spark models

## 10. Scalability Analysis

Two experiments are run in `3_model_training.ipynb`:

**Strong Scaling** – fixed 200,000 rows, vary partitions (2/4/8/16):
- Ideal: $T(n) = T(2) / (n/2)$
- Measures parallel efficiency

**Weak Scaling** – rows proportional to partitions (50,000 × n_parts):
- Ideal: constant wall-clock time
- Measures memory/shuffle overhead

Results are saved as:
- `data/samples/strong_scaling.csv`
- `data/samples/weak_scaling.csv`
- `data/samples/scaling_analysis.png`

## 11. Tests

```bash
# Run all tests
python -m pytest project/tests/test_pipeline.py -v

# Or directly
python project/tests/test_pipeline.py
```

Test categories:

| Class | What is tested |
|---|---|
| `TestSmokeEndToEnd` | Tiny synthetic data through full ML pipeline |
| `TestEventCategoryAggregator` | Custom transformer output correctness |
| `TestDataIngestion` | Schema, row count, null checks on real Parquet |
| `TestFeatureEngineering` | Split existence, feature dim, label preservation |
| `TestModelSerialisation` | MLlib model save/load round-trip |
| `TestEvaluationOutputs` | CSV/PNG artefact existence, metric ranges |

Smoke tests (`TestSmokeEndToEnd`) run independently without requiring the full pipeline output.
